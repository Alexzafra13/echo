¿Qué es Echo?

Tu propio Spotify, alojado en tu casa.

  • 📂 Apuntas Echo a tu carpeta de música y la escuchas desde cualquier dispositivo.
  • 🔒 Privacidad total: todo vive en tu servidor, sin terceros.
  • 🤝 Social y conectado: amigos, actividad y federación entre servidores.
  • 🎚️ Inteligente: listas algorítmicas y modo DJ con análisis de audio.
A diferencia de Navidrome o Jellyfin, Echo no es solo funcional: es social y conectado.
La idea es sencilla: es tu propio Spotify, pero en tu casa, respetando tu privacidad. Lo que lo diferencia de otras alternativas self-hosted es la dimensión social y la federación.

El problema que resuelve

❌ Servicios comerciales

  • No controlas tu biblioteca
  • Las canciones aparecen y desaparecen
  • Venden tus hábitos de escucha
  • No tienen tu música rara / FLAC

❌ Alternativas self-hosted

  • Funcionales pero "frías"
  • No permiten compartir
  • No conectan instancias
  • Sin descubrimiento inteligente
✅ Echo cubre ese hueco: propiedad + privacidad + lo social.

Arquitectura hexagonal

Puertos y adaptadores · los 25 módulos siguen el mismo patrón

Regla de oro

El dominio NO depende de la infraestructura. Depende de interfaces.

  • Presentación — Controllers, DTOs
  • Dominio — Entidades, casos de uso, puertos
  • Infraestructura — Repos Drizzle, caché, APIs
Ventajas: testabilidad (repos en memoria), intercambiabilidad (caché como decorador) y orden a escala.
Esta es la parte técnica clave. La arquitectura hexagonal separa la lógica de negocio de la infraestructura. El dominio depende de interfaces, los puertos, no de implementaciones concretas.

El dominio depende de un puerto (interfaz)


export interface IAlbumRepository {
  findById(id: string): Promise<Album | null>;
  create(album: Album): Promise<Album>;
}
// Token para la inyección de dependencias de NestJS
export const ALBUM_REPOSITORY = 'IAlbumRepository';

El caso de uso recibe la abstracción inyectada


@Injectable()
export class GetAlbumUseCase {
  constructor(
    @Inject(ALBUM_REPOSITORY)
    private readonly albumRepository: IAlbumRepository,
  ) {}

  async execute(input: GetAlbumInput): Promise<GetAlbumOutput> {
    const album = await this.albumRepository.findById(input.id);
    if (!album) throw new NotFoundError('Album', input.id);
    return { id: album.id, name: album.name /* ... */ };
  }
}
Fijaos: el caso de uso no conoce PostgreSQL. Solo conoce la interfaz IAlbumRepository. En tests le inyecto un repositorio en memoria; en producción, el de Drizzle. Y la caché es otro adaptador que implementa la misma interfaz.

Veamos las cuatro más interesantes técnicamente →

📡 Escáner de biblioteca

  • Recorrido recursivo con detección de symlinks y bucles.
  • Extrae metadatos con FFprobe + music-metadata.
  • Vigilancia en vivo (chokidar) con debounce de 5s.
  • Corre en una cola BullMQ y emite progreso por WebSocket.
Clave: escanear miles de canciones nunca bloquea la API → la petición responde 202 y el trabajo va a background.
El escaneo de una biblioteca grande podría tardar minutos. Por eso no se hace en la petición HTTP: se encola en BullMQ y un worker lo procesa emitiendo progreso en tiempo real.

🎵 Streaming de audio

  • Peticiones por rango HTTP (Range → 206 Partial Content): permite buscar dentro de la canción sin descargarla entera.
  • Stream tokens: token de un solo uso, hasheado (SHA-256) en BD; la URL de audio no expone el JWT.
  • Defensa path traversal: doble validación de ruta + fs.stat().
Seguridad y eficiencia de streaming a nivel de servicio comercial.

🌊 Recomendaciones — Wave Mix


score = 0.30·explícito + 0.50·implícito + 0.18·recencia + 0.02·diversidad
  • Explícito: tus valoraciones · Implícito: reproducciones + finalización
  • Recencia: decaimiento exponencial · Diversidad: evita saturar un artista
  • Balance temporal: 40% semana / 30% mes / 20% año / 10% antiguo → redescubre joyas olvidadas
Optimización: en vez de 3 consultas por canción (3·N), precarga en 3 consultas y puntúa en memoria → de O(3N) a O(N).
El Wave Mix puntúa tus canciones con una fórmula ponderada y luego equilibra por antigüedad para que no salga siempre lo mismo. Y lo hace de forma eficiente, evitando el problema N+1.

🎚️ Modo DJ + análisis de audio

  • Analiza cada canción con Essentia.js (WebAssembly): BPM, tonalidad, energía, bailabilidad.
  • Mezcla armónica con la rueda Camelot (24 tonalidades), como Rekordbox/Serato.
  • Pool de workers (núcleos÷2) con reciclado cada 250 análisis para controlar la memoria del WASM.
  • Fallback a FFmpeg si Essentia no está disponible.
Encadena temas armónicamente compatibles automáticamente, como un DJ real.
Esta es probablemente la funcionalidad más técnica. Uso teoría musical real: la rueda Camelot que usan los DJs. El reto fue la memoria de WebAssembly, resuelto con un pool de workers reciclables.

Reproduce música de la biblioteca de otra persona, con su permiso y control.

🔐 Seguridad

  • bcrypt (coste 12) para contraseñas
  • Mitigación de timing attacks (hash dummy)
  • JWT access (24h) + refresh (7d) con jti
  • Blacklist de refresh tokens (anti-replay)
  • Roles: JwtAuthGuard + AdminGuard
  • Stream tokens hasheados
  • Defensa path traversal
  • Rate limiting (HTTP + WebSocket) · Helmet
La seguridad se cuida en varias capas: contraseñas con bcrypt, tokens JWT con revocación, tokens de streaming separados y hasheados, protección de rutas y rate limiting.

⚡ Rendimiento y asincronía

Caché (cache-aside)

  • Redis con TTL por tipo de dato
  • Álbum 1h · Artista 2h · Búsqueda 1min
  • Caché como decorador transparente

Colas BullMQ

  • Escaneo, LUFS, análisis DJ
  • Concurrencia adaptada a la CPU
  • Nunca bloquean la API
Las operaciones caras (escaneo, audio, APIs externas) nunca degradan la experiencia del usuario.

✅ Calidad: testing y CI/CD

  • 374 archivos de test (Jest backend · Vitest frontend): unitarios, integración y e2e.
  • GitHub Actions: lint + tests + build con PostgreSQL y Redis reales en cada PR.
  • Husky + lint-staged: formato automático antes de cada commit.
  • Swagger/OpenAPI autogenerado en /api/docs.
  • Lighthouse CI para auditar rendimiento del frontend.
La arquitectura hexagonal hace que probar el dominio sea fácil y rápido.
